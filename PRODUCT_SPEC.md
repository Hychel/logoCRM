PRODUCT SPECIFICATION (PRD) — LogoCRM
1. Project Overview
LogoCRM — це спеціалізована веб-орієнтована CRM-система, розроблена для автоматизації внутрішніх процесів логопедичних центрів та приватних кабінетів. Платформа об'єднує в єдиному цифровому просторі роботу адміністраторів, практикуючих логопедів та керівників центру, автоматизуючи ведення бази пацієнтів, інтерактивний розклад занять, фіксацію динаміки корекції мовлення та базовий фінансовий облік.
2. Core Problem & Solution
Problem: Менеджмент у логопедичних центрах часто страждає від хаотичного ведення записів у паперових блокнотах або негнучких Excel-таблицях, що призводить до накладок у розкладі кабінетів та втрати клієнтів. Крім того, логопеди витрачають робочий час на заповнення рукописних карток розвитку, через що важко оперативно відстежувати довгостроковий прогрес пацієнта (наприклад, постановку звуків), а керівництво не має прозорої картини щодо завантаженості персоналу та реальних доходів/витрат.
Solution: LogoCRM централізує та структурує всі дані центру. Система надає інтерактивний календар для миттєвого керування розкладом, цифрові «Картки розвитку пацієнта» з історією кожного візиту та вбудований модуль обліку фінансів (абонементи, разові оплати), що звільняє час спеціалістів для безпосередньої роботи з пацієнтами та виключає помилки адміністрування.
3. Target Audience & Roles
Director / Owner (System Admin):
- керувати розкладом усіх спеціалістів;
- створювати та редагувати картки пацієнтів;
- записувати пацієнтів на заняття;
- змінювати статуси занять (проведено, скасовано, перенесено);
- приймати оплату та керувати абонементами;
- переглядати фінансову аналітику та звіти;
- бачити статистику завантаженості кабінетів і спеціалістів;
- керувати обліковими записами персоналу.
Authorized Speech Therapist (Specialist): Має доступ до власного розкладу занять та карток пацієнтів, закріплених за ним. Може заповнювати текстові нотатки після занять, відстежувати динаміку мовленнєвого прогресу конкретної дитини/дорослого та призначати домашні завдання. Не бачить фінансову звітність центру та розклад інших колег.
5. Key Features (MVP)
Authentication & Role-Based Access Control (RBAC): Захищений вхід у систему за логіном/паролем із автоматичним перенаправленням користувача на інтерфейс відповідно до його ролі (на основі сесій або JWT-куків).
Interactive Schedule & Calendar: Візуальна сітка розкладу (день/тиждень) з колірним кодуванням статусів занять. Реалізується за допомогою JS-бібліотеки FullCalendar, що інтегрується безпосередньо в HTML-сторінку.
Digital Patient Card (Speech Therapy CRM): Електронна медична картка пацієнта, що містить загальну інформацію (ПІБ, вік, контакти батьків), первинний діагноз/запит, а також хронологічну стрічку візитів із нотатками логопеда, фіксацією поставлених звуків та домашніми завданнями.
Finance & Subscription Manager: Модуль для обліку оплат. Дозволяє фіксувати разові платежі або продавати абонементи з автоматичним списуванням балансу після того, як заняття відзначається як «проведене».
Data Seeding & Mock Data Layer: Бекенд автоматично наповнює базу даних (seed) тестовими даними для демонстрації: 3 логопеди, 5 кабінетів, 20 пацієнтів з історією занять та згенерованим розкладом на поточний тиждень.
6. Technical Architecture & Constraints
Frontend Architecture & UI (Server-Side Rendering)
Інтерфейс додатку інтегрований безпосередньо в Python-проект. Рендеринг сторінок відбувається на сервері за допомогою шаблонізатора Jinja2. Клієнтська логіка (динамічні модальні вікна, робота з календарем та AJAX-запити) пишеться на чистому JavaScript (Vanilla JS) без використання важких фреймворків.
Styling & Design: Стилізація сторінок виконується через Tailwind CSS, який підключається через CDN (для етапу MVP) або компілюється локально. Дизайн адаптивний — інтерфейс коректно відображається як на моніторах ПК адміністраторів, так і на планшетах логопедів.
Interactive Components: Календар розкладу будується за допомогою бібліотеки FullCalendar.js через пряме підключення скрипта в HTML. Плавні переходи та динамічне оновлення даних (наприклад, швидкий запис на заняття без перезавантаження сторінки) реалізуються через стандартний fetch() API у JavaScript.
Unified Backend & Frontend Project Structure
Весь проект знаходиться в одному репозиторії та має наступну структуру:
Plaintext
logocrm/
├── app/
│   ├── core/              # Конфігурація (config.py), безпека (security.py)
│   ├── database/          # Session, Base моделі, скрипти Data Seeding

│   ├── schemas/           # Pydantic моделі для валідації API-запитів та форм
│   ├── repositories/      # Шаблон Repository для прямих запитів до бази даних
│   ├── services/          # Бізнес-логіка (обчислення абонементів, перевірка накладок)
│   ├── routers/           # Контролери FastAPI (віддають HTML сторінки або JSON для AJAX)
│   │   ├── views.py       # Маршрути, які рендерять HTML (Home, Calendar, Patients)
│   │   └── api_v1.py      # Маршрути для AJAX запитів (оновлення статусів, швидкий запис)
│   ├── static/            # Статичні файли сайту
│   │   ├── css/           # Стилі (tailwind.css)
│   │   └── js/            # Скрипти (main.js, calendar.js для ініціалізації календаря)
│   ├── templates/         # HTML-шаблони Jinja2
│   │   ├── base.html      # Головний шаблон (шапка, сайдбар, підключення Tailwind)
│   │   ├── login.html     # Сторінка входу
│   │   ├── calendar.html  # Сторінка з інтерактивним календарем
│   │   ├── patients.html  # Список пацієнтів та картка пацієнта
│   │   └── dashboard.html # Панель директора з фінансовими звітами
│   └── main.py            # Точка входу в додаток FastAPI
├── migrations/            # Файли міграцій бази даних Alembic
└── requirements.txt       # Залежності (fastapi, uvicorn, sqlalchemy, jinja2, alembic)
Database & ORM
Database: Реляційна СУБД PostgreSQL, яка забезпечує цілісність даних при зв'язках типу «Пацієнт — Абонемент — Заняття».
ORM & Migrations: SQLAlchemy (Asyncio) як ORM-система для побудови типізованих запитів. Керування змінами структури бази даних здійснюється через Alembic.
Validation & Security
Валідація даних, що надходять з HTML-форм (або через AJAX), реалізується через Pydantic на рівні шару schemas.
Автентифікація реалізована за допомогою JWT (JSON Web Tokens) або сесійних кук. Токен безпечно зберігається в HttpOnly Cookies, що повністю захищає додаток від XSS-атак. Паролі хэшуються за допомогою passlib (bcrypt).
Local Development Environment Network Ports
Оскільки проект є монолітним, весь додаток (і логіка, і інтерфейс) працює на одному порті:
URL додатку (FastAPI + HTML/CSS): http://localhost:5000
Interactive Swagger UI (для тестування API): http://localhost:5000/docs
Infrastructure & Deployment
Database Layer: Дві ізольовані бази даних у хмарі (наприклад, на Supabase або Neon): logocrm-db-dev та logocrm-db-prod.
Hosting: Додаток (FastAPI разом із папками static та templates) упаковується в один Docker-контейнер і розгортається на Azure App Services (Linux Plan) або Render/Railway.
CI/CD: Налаштовується через GitHub Actions. Конвеєр автоматично запускає тести при кожному пуші. Деплой на Production відбувається автоматично після успішного проходження тестів або створення релізного тегу. Всі секрети (ключі шиврування, рядки підключення до БД) передаються через змінні оточення (Environment Variables).
7. System Design Guidelines for AI
Код має бути організованим і зрозумілим, з розділенням логіки між backend, frontend та роботою з базою даних.
Робота з базою даних повинна виконуватись через ORM-моделі та окремі модулі для доступу до даних.
Бізнес-логіку бажано виносити в окремі сервіси або helper-функції.
Структура проєкту повинна залишатися чистою, масштабованою та зручною для подальшої підтримки.
API та сторінки повинні бути розділені на окремі маршрути для зручності підтримки системи.
